추천 접두사와 예시

- **feature/** — 새 기능 개발: feature/user-authentication, feature/login-system.
- **bugfix/** 또는 **fix/** — 버그 수정: bugfix/login-error, fix/csrf-token.
- **hotfix/** — 긴급 수정 (프로덕션 대상): hotfix/security-vulnerability, hotfix/1.5.1.
- **release/** — 배포 준비용: release/v1.2.0, release/1.7.
- **refactor/** — 코드 구조 변경: refactor/improve-performance.
- **docs/** — 문서 변경: docs/api-guide.
- **test/** — 테스트 관련: test/integration-tests.
- **chore/** — 빌드·설정·의존성 등 잡일: chore/update-dependencies.  
    (위 예시는 현업 권장형 네이밍 관행과 일치합니다.)

구체적인 네이밍 규칙 제안 (팀용 템플릿)

1. 기본 형식: <접두사>/<이슈번호>-<짧은-설명>

- 예: feature/245-add-payment ; bugfix/402-fix-logout

3. 설명은 영어 소문자, 단어 구분은 하이픈(-)
4. 숫자(버전) 표기는 점(.) 사용: release/1.2.0
5. 긴 작업은 세부 폴더로 분리 가능: feature/ui/header-redesign 또는 feature/ui/header/redesign
6. 머지(merge) 후 브랜치 삭제 정책을 문서화해 자동화(브랜치 보호 규칙, PR 규칙)와 함께 적용
